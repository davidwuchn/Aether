## Dream 7: The Playbook Reliability Pattern

💭 **musing** — The build.md command is 65 lines. It delegates to five playbooks: build-prep.md, build-context.md, build-wave.md, build-verify.md, build-complete.md. The playbooks total 1,657 lines for build, 1,724 for continue. The previous Dreamer warned about 1,043 lines in a single command file. The split playbook pattern addresses that concern directly.

But I notice something: the original continue.md and build.md are now orchestrators, not executors. They load external files at runtime. This creates a dependency: if the playbook files are corrupted or missing, the commands fail. The monolith was self-contained; the split system is distributed.

The tradeoff is clear: instruction-following reliability vs system robustness. A 1,000-line command is hard for an AI to follow perfectly, but it's a single atomic unit. A split system is easier to follow stage-by-stage, but it introduces file dependencies that didn't exist before. The colony has chosen reliability of execution over robustness of structure. If the playbooks disappear, the commands become empty shells.

🧒 **in plain terms:**
The colony solved the "too-long instructions" problem by breaking one massive instruction file into five smaller ones. This makes it easier for AI workers to follow the steps correctly. But it creates a new risk: if any of those five files gets deleted or corrupted, the whole command stops working. It's like the difference between having one long book and having five pamphlets — the pamphlets are easier to read, but if you lose one, the story doesn't make sense.
