## Plan A: Gate Failure Recovery

When Aether's verification gates fail (Auditor, Gatekeeper, Watcher Veto, etc.), the user gets a big red "FAILED" banner and the process stops dead. The recovery path exists (fix issues, run /ant-continue again) but it's buried in small text. The Watcher Veto is especially destructive — it auto-stashes all your work, making you feel like you have to start over. There's no agent to help fix gate failures, and re-running /ant-continue checks ALL gates from scratch even if most already passed.

This change makes gate failures recoverable, fixable, and clear.

## Plan B: Smart Review Depth — Auto, Light, and Final-Phase Override

Build and continue currently spawn a heavy review gauntlet — Auditor, Gatekeeper, Probe, Weaver, Medic, Measurer, Chaos — on every phase. For most phases this is overkill.

### Review Depth Modes

**Light (most phases — default):**
- Skipped: Auditor, Gatekeeper, Probe, Weaver, Medic, Measurer, Chaos
- Kept: Watcher, build/type/lint/test commands, anti-pattern scan, TDD evidence, flags gate, Watcher veto, runtime verification

**Heavy (final phase + security-sensitive phases):**
- Full review gauntlet — current behavior, unchanged.

### Auto-Detection Logic
1. --light flag present → light (user override)
2. Phase is the last phase → heavy (always, non-negotiable)
3. Phase name contains security keywords → heavy
4. Phase name contains release keywords → heavy
5. Default → light

2. cmd/codex_workflow_cmds.go — --light flags on build and continue
3. cmd/codex_build.go — Wire review depth to skip optional workers

## Plan D: Add the Porter Ant (Interactive Delivery Caste)

Aether colonies build things but have no ant responsible for shipping them. The Porter ant is a new caste that gets WIRED INTO the lifecycle — it's not a command you run manually. It prompts the user with multiple choice options at key moments (seal, build completion, etc.).

### Key Design Principle
Porter is interactive and wired-in, NOT a standalone CLI command you run. It surfaces as prompts:
- "Do you want to push this to remote?"
- "Do you want to publish to the package registry?"
- "Do you want to deploy?"

The user picks, and Porter executes.

### Steps
1. **Go Runtime — Caste Registration:** Add porter to casteEmojiMap ("📦"), casteColorMap ("95"), casteLabelMap, castePrefixes, codexAgentNameForCaste
2. **Go Runtime — Porter Logic:** cmd/porter_cmd.go with check/publish/status — but primarily called from lifecycle hooks, not directly
3. **Seal Integration — Interactive Porter Prompt:** After seal completes, Porter surfaces interactive options for publishing/pushing/deploying (like how seal already has interactive elements)
4. **Claude Code Agent:** .claude/agents/ant/aether-porter.md — handles the interactive prompting and execution
5. **Packaging Mirrors:** agents-claude, opencode, codex (TOML)

### Porter Subcommands (backend, mainly lifecycle-called)

## Plan F: Idea Shelving — Guided Milestone Continuity

Right now the next milestone is freestyled — the user has to think "what's the next init command?" There should be a shelving system where ideas get captured during colony work, stored, and automatically promoted to future colonies when relevant.

### Concept
- During colony work, workers or users spot good ideas that don't fit the current scope
- These ideas get "shelved" — stored in a persistent file (like a colony backlog)
- When a new colony starts (via /ant-init or /ant-seal), the system surfaces relevant shelved ideas
- This creates continuity between colonies — the system guides the user, not the other way around

### Key Features
- Shelved ideas live in a persistent file (colony-scoped, survives entomb)
- Ideas can have trigger conditions (e.g., "surface when starting a new colony in this repo")
- /ant-seal automatically shelves promising but unimplemented ideas
- /ant-init surfaces relevant shelved ideas and lets user promote or defer them
- Integration with pheromones: REDIRECT signals that keep recurring get shelved as permanent guidance

The global QUEEN.md (~/.aether/QUEEN.md) is broken in multiple ways. It has ~270 identical `<repo> wisdom` lines. But the bigger problem is the pipeline itself — content isn't flowing properly.

### Root Causes (Data)

2. No dedup in appendEntriesToQueenSection (cmd/queen.go:512): every time queen-seed-from-hive runs, it blindly appends all entries
3. queen-seed-from-hive has no dedup guard (cmd/queen.go:298-351)
4. The abstraction step replaces repo names with `<repo>`, making test entry literally `<repo> wisdom` — useless placeholder

### Pipeline Wiring Gaps (discovered via code research)
1. **Global wisdom not injected into workers**: colony-prime only injects LOCAL QUEEN.md wisdom, not global (~/.aether/QUEEN.md). The global Wisdom and Patterns sections are read by readQUEENMd() but never make it into worker prompts.
2. **Philosophies and Anti-Patterns sections unused**: Only Wisdom and Patterns sections are extracted for injection. Philosophies, Anti-Patterns, and Colony Charter are completely ignored by colony-prime.
3. **queen-promote-instinct only writes local**: Promoted instincts go to local QUEEN.md only, never to global. They're trapped in one repo.
4. **No automatic instinct promotion**: High-confidence instincts are NOT automatically promoted to QUEEN.md — they require explicit commands.

### Plan
1. Clean hive wisdom data — remove test entry from ~/.aether/hive/wisdom.json
2. Clean QUEEN.md — remove all `<repo> wisdom` duplicate lines, keep real content
3. Add dedup to appendEntriesToQueenSection — check if entry already exists before appending
4. Add dedup guard to queen-seed-from-hive — filter entries already in QUEEN.md
5. Fix colony-prime injection — inject global QUEEN.md wisdom alongside local wisdom
6. Inject Philosophies and Anti-Patterns sections into worker context
7. Fix queen-promote-instinct to write to global QUEEN.md (not just local)
8. Add automatic instinct promotion at /ant-seal for confidence >= 0.8
9. Add test coverage for dedup behavior and pipeline wiring
