# Phase 56: Lifecycle Integration - Research

**Researched:** 2026-04-26
**Domain:** Go CLI lifecycle command integration (seal, entomb, status, init)
**Confidence:** HIGH

## Summary

Phase 56 integrates the review ledger system (built in phases 53-55) into the four colony lifecycle commands: seal, entomb, status, and init. Each integration point is a small, surgical modification to an existing, well-structured command. The review ledger data types (`ReviewLedgerFile`, `ReviewLedgerSummary`, `ReviewLedgerEntry`) and the `review-ledger-summary` subcommand already exist and can be reused directly.

The seal command currently writes a `CROWNED-ANTHILL.md` summary but does NOT archive any data directories (survey, build, reviews). The requirement LIFE-01 says "archives alongside existing survey and build archives" but no such archiving currently exists in seal. The implementation needs to ADD the reviews archival step to seal (and potentially clarify whether survey/build archival is also new or was already planned).

The entomb command already has a well-established pattern for copying data into chamber archives via `copyEntombArtifacts` and `copyDirIfExists`. Adding the reviews directory follows this pattern exactly.

The status command renders a rich dashboard via `renderDashboard`. Review ledger counts fit naturally after the existing "Memory Health" section or alongside the pheromone summary.

The init command cleans up stale state during colony creation. It currently cleans worktrees but does not touch review data. Adding `os.RemoveAll` of the reviews directory follows the same cleanup pattern.

**Primary recommendation:** Each requirement maps to a small, independent change in an existing command file. Reuse `review-ledger-summary` logic for reading domain counts. Follow existing patterns (`copyDirIfExists`, `os.RemoveAll`, pretty table rendering). No new packages or dependencies needed.

### Pattern 1: Seal Review Archival (LIFE-01, LIFE-02)
**What:** Before writing CROWNED-ANTHILL.md, scan all domain ledgers for open HIGH-severity entries. Include warnings in the seal summary. Then copy the reviews directory to an archive location.
**When to use:** When seal needs to surface unresolved high-severity findings.
**Example:**
```go
// In sealCmd.RunE, after state validation but before writing CROWNED-ANTHILL.md:

// Scan for high-severity open findings
highSeverityWarnings := []string{}
for _, d := range colony.DomainOrder {
    ledgerPath := fmt.Sprintf("reviews/%s/ledger.json", d)
    var lf colony.ReviewLedgerFile
    if err := store.LoadJSON(ledgerPath, &lf); err != nil {
        continue
    }
    for _, e := range lf.Entries {
        if e.Status == "open" && e.Severity == colony.ReviewSeverityHigh {
            highSeverityWarnings = append(highSeverityWarnings, fmt.Sprintf(
                "[%s] %s: %s", d, e.ID, e.Description,
            ))
        }
    }
}

// Add warnings to seal summary if any exist
if len(highSeverityWarnings) > 0 {
    // Append to CROWNED-ANTHILL.md content
}

// Archive reviews directory (LIFE-01)
// Follow the pattern used by entomb's copyDirIfExists
```

### Anti-Patterns to Avoid
- **Do NOT read ledger files directly in status without using storage.Store**: The store handles path resolution and file locking. Use `store.LoadJSON("reviews/security/ledger.json", &lf)`.
- **Do NOT create a new "archive" concept for seal**: The roadmap says "archives alongside existing survey and build archives" but seal currently does NOT archive survey or build directories. Only add review archival, not a new general archival system.
- **Do NOT modify review ledger data types**: The types from Phase 53 are stable and should not change. Reuse them as-is.
- **Do NOT add review logic to colony-prime**: That was Phase 54's job and is already done. Phase 56 only touches lifecycle commands.

## Common Pitfalls

### Pitfall 1: Seal "Archives Alongside" Misinterpretation
**What goes wrong:** The requirement says "archives alongside existing survey and build archives" but seal currently does NOT archive any directories. A naive reading would try to find and mimic non-existent survey/build archival code.
**Why it happens:** The requirement language implies prior art that does not exist.
**How to avoid:** Treat LIFE-01 as "add reviews directory archival to seal." The "alongside" language is aspirational, not referring to existing code. Copy the reviews directory to a location adjacent to CROWNED-ANTHILL.md (e.g., `.aether/data/reviews-archive/` or simply ensure the reviews dir is not cleaned until entomb handles it).
**Warning signs:** Spending time looking for `survey` or `build` archival code in seal.

### Pitfall 2: Init Cleanup Race Condition
**What goes wrong:** Init removes the reviews directory, but if the store has already been created and the reviews path is being accessed concurrently, file operations could fail.
**Why it happens:** Init runs before the new store is saved, so this is safe in practice, but the order of operations matters.
**How to avoid:** Place the `os.RemoveAll` for reviews AFTER the `gcOrphanedWorktrees()` call but BEFORE creating the new `COLONY_STATE.json`. This matches the existing cleanup pattern.
**Warning signs:** Placing cleanup after `store.SaveJSON("COLONY_STATE.json", state)`.

### Pitfall 3: Status Rendering With No Reviews
**What goes wrong:** Status dashboard shows an empty table or "0 findings" for all 7 domains even when no review data exists, cluttering the output.
**Why it happens:** Colony-prime already handles this with "omit entirely when no open findings" (PRIME-04). Status should follow the same pattern.
**How to avoid:** Only render the Review Findings section if at least one domain has entries. Skip the section entirely if no ledgers exist.
**Warning signs:** Adding a "Review Findings" header that appears on every status call regardless of data.

### Pitfall 4: Entomb Verification Missing Reviews
**What goes wrong:** The `verifyEntombedChamber` function checks for required files in the chamber. If reviews are now expected but the test was created before reviews existed, the chamber might be rejected.
**Why it happens:** The verification function has a hard-coded list of required files.
**How to avoid:** Do NOT add reviews to the required files list in `verifyEntombedChamber`. Reviews are optional -- a colony may have been sealed before Phase 56 and have no review data. The existing required list should remain unchanged.
**Warning signs:** Adding `filepath.Join(chamberDir, "reviews")` to the `required` slice in `verifyEntombedChamber`.

## Code Examples

### Scanning for High-Severity Open Findings (LIFE-02)
```go
// Reuse the pattern from buildPriorReviewsSection in colony_prime_context.go
func scanHighSeverityOpen(s *storage.Store) []string {
    var warnings []string
    for _, d := range colony.DomainOrder {
        ledgerPath := fmt.Sprintf("reviews/%s/ledger.json", d)
        var lf colony.ReviewLedgerFile
        if err := store.LoadJSON(ledgerPath, &lf); err != nil {
            continue
        }
        for _, e := range lf.Entries {
            if e.Status == "open" && e.Severity == colony.ReviewSeverityHigh {
                loc := ""
                if e.File != "" {
                    loc = e.File
                    if e.Line > 0 {
                        loc = fmt.Sprintf("%s:%d", e.File, e.Line)
                    }
                }
                if loc != "" {
                    warnings = append(warnings, fmt.Sprintf("- [%s] %s: %s (%s)", d, e.ID, e.Description, loc))
                } else {
                    warnings = append(warnings, fmt.Sprintf("- [%s] %s: %s", d, e.ID, e.Description))
                }
            }
        }
    }
    return warnings
}
```

### Status Review Findings Table (LIFE-04)
```go
// Follow renderMemoryHealthTable pattern from status.go
func renderReviewFindingsTable(b *strings.Builder, s *storage.Store) {
    t := table.NewWriter()
    t.AppendHeader(table.Row{"Domain", "Total", "Open", "Resolved"})

    anyData := false
    for _, d := range colony.DomainOrder {
        ledgerPath := fmt.Sprintf("reviews/%s/ledger.json", d)
        var lf colony.ReviewLedgerFile
        if err := s.LoadJSON(ledgerPath, &lf); err != nil {
            continue
        }
        if lf.Summary.Total == 0 {
            continue
        }
        anyData = true
        t.AppendRow(table.Row{d, lf.Summary.Total, lf.Summary.Open, lf.Summary.Resolved})
    }

    if !anyData {
        b.WriteString("   No review findings recorded\n")
        return
    }

    t.SetStyle(table.StyleRounded)
    b.WriteString(t.Render() + "\n")
}
```

## Open Questions (RESOLVED)

1. **Should seal archive reviews to a separate location, or just ensure they persist until entomb?** (RESOLVED)
   - **Resolution:** Seal copies the reviews directory to `reviews-archive/` alongside CROWNED-ANTHILL.md. This satisfies LIFE-01's "archives" requirement while keeping entomb as the full archival path. Seal does NOT delete or move the original reviews directory, leaving it for entomb to archive into the chamber.

2. **Should the status Review Findings section show severity breakdowns or just total/open?** (RESOLVED)
   - **Resolution:** Show Domain, Total, Open, Resolved columns. No severity breakdown column. The full severity breakdown is available via `review-ledger-summary` for users who need it. Keeping the status table simple with four columns avoids cluttering the dashboard.
