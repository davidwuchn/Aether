<objective>
Add gate result persistence, per-gate recovery templates, incremental skip logic, and CLI subcommands to the Go runtime layer.

Purpose: When gates fail, users get specific recovery instructions (not just "FAILED"). When they re-run continue, passed gates are skipped so only failures are re-checked. This plan creates the Go runtime foundation AND exposes it via cobra CLI subcommands so the playbook layer (Plan 02) can call `aether gate-results-read`, `aether gate-results-write`, etc.
Output: GateResultEntry type, recovery templates map, gate results read/write helpers, skip logic, gate results threading in the continue command, and 4 cobra CLI subcommands.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Go runtime -> COLONY_STATE.json | Gate results are written to local state file; no external input crosses this boundary |
| priorGateResults input | Read from COLONY_STATE.json which is user-modifiable; a user could manually set all gates to passed |
| CLI subcommands | Exposed via `aether` binary; playbook layer calls these -- CLI args are validated (--name required) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-59-01 | Tampering | gate_results in COLONY_STATE.json | accept | User controls their own state file; self-hosted tool, not a security boundary. Tests always re-run regardless (D-10) which catches stale results |
| T-59-02 | Information disclosure | Recovery templates | accept | Templates are hardcoded in binary, not user-controlled. No PII involved |
| T-59-03 | Denial of service | gateResultsWrite on every gate | accept | Write is atomic and fast (< 1ms). At most 14 gates per continue run. No amplification vector |
| T-59-04 | Tampering | CLI --name flag injection | accept | Flag value is a gate name string; not used in shell execution or SQL. Low-value target |
</threat_model>
